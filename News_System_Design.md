# 뉴스 검색 시스템 설계 초안 — Agentic RAG 기반 2단계 Thinking Depth 뉴스 시스템

---

## 새 시스템의 목적

새 시스템의 목표는 투자자마다 정보를 처리하는 깊이(Thinking Depth)가 다르다는 현실을 반영하는 것이다. 모든 에이전트는 전처리 단계에서 선별된 **일일 뉴스 제목 10개**를 동일한 출발점으로 받는다. 다만 이 제목들을 어떻게 처리하는지는 각 에이전트의 Thinking Depth에 따라 달라진다.

새 시스템에서는 제목만 보고 넘어가는 유형과 관심 있는 뉴스 일부를 읽는 유형을 별도 Depth로 엄격하게 나누지 않는다. 두 행동은 모두 기본적인 투자자 행동 안에 포함되며, 에이전트가 자신의 Belief, 계좌 상태, 투자 성향, 뉴스 제목의 중요도를 바탕으로 **읽을지 말지를 자율적으로 판단**한다.

따라서 기본형 에이전트는 일일 뉴스 제목 10개를 확인한 뒤, 필요 없다고 판단하면 제목만 참고하고 바로 매매 결정 단계로 넘어간다. 필요하다고 판단하면 관심 있는 뉴스 일부를 골라 `read_news` 도구로 내용을 읽는다. 심층형 에이전트는 여기에서 더 나아가, 뉴스를 읽은 뒤 추가로 궁금한 분야를 정하고 최근 7일 뉴스에서 `search_news` 도구를 통해 더 깊게 탐색한다.

기술적으로는 Agentic RAG를 기반으로 한다. 에이전트가 도구를 사용하여 뉴스 데이터베이스와 직접 상호작용하고, 필요에 따라 뉴스 읽기와 추가 검색을 수행하는 구조다. 검색 방식은 단순 의미적 유사도(semantic similarity)에만 의존하지 않고, 키워드 기반 직접 검색 도구를 에이전트에게 제공함으로써 더 정확하고 통제 가능한 검색을 목표로 한다. 이는 "Beyond Semantic Similarity: Rethinking Retrieval for Agentic Search via Direct Corpus Interaction" 논문의 문제의식과 맥락을 같이 한다.

## 핵심 설계 1 — Persona에 Thinking Depth 내재화

Thinking Depth는 에이전트의 Persona에 속성으로 할당되며, **1, 2 두 단계로만 구분한다**. Depth의 분포는 에이전트 초기화 시점에 결정되며, 이후 시뮬레이션 전반에 걸쳐 고정된다. Depth 배분 비율은 실험 파라미터로 조정 가능하게 설계한다.

Depth 페르소나는 다음 원칙에 따라 적용한다.

- **Depth 1은 기본 투자자 Persona 안에 적용한다.**  
  Depth 1은 일일 뉴스 제목 10개를 보고 판단하되, 관심 있는 뉴스 일부를 읽을지 여부는 에이전트가 최대한 자율적으로 결정한다. 즉, 뉴스 본문을 전혀 읽지 않을 수도 있고, 필요하다고 판단한 뉴스만 제한적으로 읽을 수도 있다.

- **Depth 2만 별도의 심층 탐색 Persona를 추가 적용한다.**  
  Depth 2는 단순히 뉴스를 읽는 수준을 넘어, 읽은 뉴스에서 파생된 궁금증을 바탕으로 검색 분야를 정하고 키워드를 구성한 뒤 추가 검색을 수행하는 에이전트로 설계한다.

### Depth 1

Depth 1은 기본적인 뉴스 처리 유형이다. 이 에이전트는 매일 제공되는 뉴스 제목 10개를 확인한 뒤, 제목만으로 판단할지, 관심 있는 뉴스 일부를 읽을지를 자율적으로 결정한다.

Depth 1의 원칙은 다음과 같다.

- 입력으로 받은 일일 뉴스 제목 10개를 우선 확인한다.
- `read_news`를 호출할지 여부는 에이전트가 판단한다.
- 읽고 싶은 뉴스가 없으면 `read_news`를 호출하지 않아도 된다.
- 읽고 싶은 뉴스가 있으면 일일 뉴스 제목 10개 안에서만 선택한다.
- `read_news`로 읽을 수 있는 뉴스는 최대 3개다.
- `search_news`를 통한 추가 검색은 수행하지 않는다.

이는 제목 수준의 시장 신호만 참고하는 투자자와, 제목을 훑은 뒤 관심 있는 기사 몇 개만 읽는 일반적인 개인 투자자를 하나의 기본 유형으로 통합한 것이다. 시스템은 읽기 여부를 강제하지 않고, 도구 사용 한도만 부여한다.

Depth 1은 다음 정보를 바탕으로 매매 결정 단계로 넘어간다.

- 현재 Belief
- 계좌 정보
- 일일 뉴스 제목 10개
- 선택적으로 읽은 뉴스 내용

### Depth 2

Depth 2는 Depth 1의 행동을 모두 포함한 뒤, 필요하면 추가 검색까지 수행하는 유형이다. 먼저 일일 뉴스 제목 10개를 확인하고, 관심 있는 뉴스를 `read_news`로 읽을지 자율적으로 판단한다. 이후 더 궁금한 내용이 있으면 최근 7일 동안의 뉴스에서 `search_news` 도구를 활용해 추가 탐색을 수행한다.

Depth 2의 추가 탐색 과정은 다음과 같다.

1. 일일 뉴스 제목과 읽은 뉴스, 현재 Belief를 바탕으로 더 궁금한 분야가 있는지 판단한다.
2. 추가 탐색이 필요하다고 판단하면 검색 분야를 정한다. 단, 검색 분야는 최대 4개로 제한한다.
3. 각 분야마다 검색에 사용할 키워드 조합을 만든다.
4. `search_news` 도구를 호출해 최근 7일 뉴스에서 분야별 관련 뉴스 제목을 받는다.
5. 받아온 뉴스 제목 중 실제로 읽고 싶은 뉴스를 골라 `read_news` 도구로 읽는다.
6. 추가 검색 결과에서 읽을 수 있는 뉴스는 최대 5개다.
7. 검색할 필요가 없거나 읽을 만한 결과가 없다고 판단하면 추가 검색 또는 추가 읽기를 생략할 수 있다.

이는 뉴스를 읽으면서 능동적으로 사고하고, 첫 정보에서 멈추지 않고 더 깊이 파고드는 투자자를 모사한다. 다만 Depth 2 역시 모든 단계를 기계적으로 수행하도록 강제하지 않고, 추가 검색과 추가 읽기의 필요성을 에이전트가 판단하도록 설계한다.

## 핵심 설계 2 — 뉴스 데이터셋 및 일일 뉴스 선정 방식

시뮬레이션은 총 6개월 동안 수행하며, 매일 다음 범주의 뉴스를 수집한다.

| 범주 | 일일 수집량 | 설명 |
|---|---:|---|
| 삼성전자 키워드 뉴스 | 50개 | 삼성전자와 직접적으로 관련된 종목 뉴스 |
| 반도체 섹터 뉴스 | 10개 | 메모리, 파운드리, HBM, AI 반도체 등 섹터 관련 뉴스 |
| 경제 관련 뉴스 | 20개 | 금리, 환율, 수출, 물가, 경기, 정책 등 거시경제 뉴스 |

수집된 뉴스는 전처리 단계를 거쳐 사용 가능한 후보 뉴스로 정리한다.

전처리 과정은 다음과 같다.

1. 중복되거나 매우 유사한 뉴스를 병합한다.
2. 삼성전자, 반도체 섹터, 경제와 상관없는 뉴스를 제외한다.
3. 하루 후보 뉴스를 약 30개 정도로 정리한다.
4. 각 뉴스를 200자 내외로 요약한다.
5. 각 뉴스의 종류를 `종목`, `섹터`, `경제` 중 하나로 분류한다.
6. 각 뉴스의 중요도를 계산한다.
7. 중요도 기준으로 매일 총 10개의 뉴스 제목을 선정한다.

일일 10개 뉴스는 다음 비율을 따른다.

| 종류 | 일일 선정 개수 |
|---|---:|
| 종목 | 5개 |
| 섹터 | 3개 |
| 경제 | 2개 |
| **합계** | **10개** |

이 구조에서는 기존의 **Impact News** 개념을 사용하지 않는다. Impact News를 별도로 선별하거나, 가격 충격 threshold를 기준으로 중요 뉴스를 사후 판정하지 않는다. 하루에 에이전트에게 제공할 10개 뉴스는 이미 데이터 전처리 단계에서 정해진 결과로 간주한다.

또한 별도의 공통 뉴스 브로드캐스트 개념도 두지 않는다. 에이전트가 매일 받는 것은 전처리 결과로 생성된 **일일 뉴스 제목 10개**이며, 이 제목 목록이 모든 Depth의 공통 출발점이 된다.

## 핵심 설계 3 — 뉴스 데이터 형식

뉴스 하나의 기본 데이터 형식은 다음과 같다.

```json
{
  "title": "뉴스 제목",
  "date": "YYYY-MM-DD",
  "content": "뉴스 내용",
  "type": "종목 | 섹터 | 경제"
}
```

각 필드의 의미는 다음과 같다.

| 필드 | 설명 |
|---|---|
| `title` | 뉴스 제목 |
| `date` | 뉴스 발행 날짜 |
| `content` | 전처리된 뉴스 내용. 기본적으로 200자 내외 요약 본문을 사용한다. |
| `type` | 뉴스 종류. `종목`, `섹터`, `경제` 중 하나다. |

일일 뉴스 제목 10개를 에이전트에게 제공할 때는 기본적으로 `title`, `date`, `type`만 노출한다. `content`는 에이전트가 `read_news` 도구를 호출했을 때만 제공한다.

## 핵심 설계 4 — Agentic Tool 기반 뉴스 읽기 및 검색

새 시스템의 도구는 크게 `read_news`와 `search_news` 두 가지로 구성된다. Depth 1은 `read_news`를 선택적으로 사용할 수 있고, Depth 2는 `read_news`와 `search_news`를 모두 사용할 수 있다.

중요한 점은 `read_news` 사용 여부를 시스템이 고정하지 않는다는 것이다. 시스템은 읽기 예산과 사용 가능한 도구만 정의하고, 어떤 뉴스를 읽을지 또는 읽지 않을지는 에이전트가 자신의 상황에 맞게 판단하도록 한다.

### 1. `read_news` 도구

`read_news`는 뉴스 제목 리스트를 입력받아, 해당 뉴스의 내용을 반환하는 도구다.

#### 입력

```json
{
  "titles": [
    "뉴스 제목 1",
    "뉴스 제목 2"
  ]
}
```

#### 출력

```json
[
  {
    "title": "뉴스 제목 1",
    "date": "YYYY-MM-DD",
    "content": "뉴스 내용 전체",
    "type": "종목"
  },
  {
    "title": "뉴스 제목 2",
    "date": "YYYY-MM-DD",
    "content": "뉴스 내용 전체",
    "type": "경제"
  }
]
```

#### 사용 제한

| Depth | 사용 가능 여부 | 제한 |
|---|---|---|
| Depth 1 | 가능 | 일일 뉴스 제목 10개 중 자율 선택, 0~3개 |
| Depth 2 | 가능 | 일일 뉴스 제목 10개 중 자율 선택 0~3개, 추가 검색 결과 중 자율 선택 0~5개 |

Depth 1의 `read_news`는 일일 뉴스 제목 10개 안에서만 사용할 수 있다. Depth 2는 Depth 1 단계에서 뉴스를 읽은 뒤, 추가 검색 결과로 받은 뉴스 제목에 대해서도 `read_news`를 사용할 수 있다.

### 2. `search_news` 도구

`search_news`는 최근 7일 동안의 뉴스에서 분야별 키워드 조합을 기준으로 관련 뉴스를 찾는 도구다. 이 도구는 Depth 2만 사용할 수 있다.

#### 입력

입력은 검색하고 싶은 분야별 키워드 조합이다. 검색 분야는 한 번의 호출에서 최대 4개로 제한하며, 하나의 분야에는 여러 개의 유사 키워드가 들어갈 수 있다.

```json
{
  "fields": [
    {
      "field": "HBM",
      "keywords": ["고대역폭메모리", "고대역폭 메모리", "AI 메모리", "HBM"]
    },
    {
      "field": "파운드리",
      "keywords": ["파운드리", "위탁생산", "첨단 공정", "2나노"]
    }
  ]
}
```

위 `fields` 배열에는 최대 4개의 분야만 입력할 수 있다.

#### 동작 방식

`search_news`는 입력된 각 분야마다 다음 과정을 수행한다. 단, 한 번에 처리할 수 있는 분야 수는 최대 4개다.

1. 현재 시점 기준 최근 7일 동안의 뉴스만 검색 대상으로 삼는다.
2. 해당 분야의 키워드들이 뉴스 제목과 내용에 얼마나 많이 등장하는지 계산한다.
3. 중복 키워드 등장 개수가 많은 뉴스일수록 높은 점수를 부여한다.
4. 점수 순서대로 정렬한다.
5. 각 분야별로 최대 5개의 뉴스를 추출한다.

#### 출력

```json
{
  "HBM": [
    {
      "title": "뉴스 제목 1",
      "date": "YYYY-MM-DD",
      "type": "종목"
    },
    {
      "title": "뉴스 제목 2",
      "date": "YYYY-MM-DD",
      "type": "섹터"
    }
  ],
  "파운드리": [
    {
      "title": "뉴스 제목 3",
      "date": "YYYY-MM-DD",
      "type": "섹터"
    }
  ]
}
```

`search_news`는 뉴스의 전체 내용을 바로 반환하지 않는다. 먼저 분야별 뉴스 제목, 날짜, 종류만 반환한다. 이후 에이전트가 실제로 읽고 싶은 뉴스를 고르면 `read_news`를 호출해 내용을 읽는다.

## 전체 Flow 설계

에이전트가 매매 결정을 내리기 전, 뉴스 정보를 처리하는 전체 과정은 다음과 같다.

### 1. 일일 뉴스 준비

시뮬레이션의 각 날짜마다 뉴스 데이터 전처리를 수행한다.

1. 삼성전자 키워드 뉴스 50개, 반도체 섹터 뉴스 10개, 경제 관련 뉴스 20개를 수집한다.
2. 중복 및 유사 뉴스를 병합한다.
3. 관련 없는 뉴스를 제외한다.
4. 후보 뉴스를 약 30개로 정리한다.
5. 각 뉴스를 200자 내외로 요약한다.
6. 중요도를 계산한다.
7. 종목 5개, 섹터 3개, 경제 2개 비율로 일일 뉴스 10개를 선정한다.

이때 선정된 일일 뉴스 10개는 모든 에이전트에게 동일하게 제공된다. 단, 에이전트에게 처음 노출되는 것은 뉴스 전문이 아니라 제목, 날짜, 종류다.

### 2. Depth별 뉴스 처리

#### Depth 1 Flow

1. 일일 뉴스 제목 10개를 확인한다.
2. 현재 Belief, 계좌 정보, 투자 성향, 뉴스 제목의 신호를 바탕으로 읽을 필요가 있는 뉴스가 있는지 판단한다.
3. 읽을 필요가 없다고 판단하면 `read_news`를 호출하지 않는다.
4. 읽고 싶은 뉴스가 있으면 일일 뉴스 제목 10개 중 최대 3개를 골라 `read_news`로 읽는다.
5. `search_news`를 호출하지 않는다.
6. 현재 Belief, 계좌 정보, 뉴스 제목 10개, 선택적으로 읽은 뉴스 내용을 바탕으로 매매 결정 단계로 넘어간다.

#### Depth 2 Flow

1. 일일 뉴스 제목 10개를 확인한다.
2. Depth 1과 동일하게 관심 있는 뉴스를 읽을지 여부를 자율적으로 판단한다.
3. 읽고 싶은 뉴스가 있으면 일일 뉴스 제목 10개 중 최대 3개를 골라 `read_news`로 읽는다.
4. 읽은 뉴스, 읽지 않은 제목 목록, 현재 Belief를 바탕으로 더 궁금한 내용이 있는지 판단한다.
5. 추가로 궁금한 내용이 있으면 검색하고 싶은 분야를 정한다. 검색 분야는 최대 4개다.
6. 각 분야마다 키워드 조합을 만든다.
7. `search_news`를 호출해 최근 7일 뉴스에서 분야별 뉴스 제목을 받는다.
8. 검색 결과로 받은 제목 중 실제로 읽고 싶은 뉴스가 있는지 판단한다.
9. 읽고 싶은 뉴스가 있으면 추가 검색 결과 중 최대 5개를 골라 `read_news`로 읽는다.
10. 추가 검색이 필요 없거나 읽을 만한 검색 결과가 없으면 추가 읽기를 생략한다.
11. 현재 Belief, 계좌 정보, 제목 목록, 읽은 뉴스, 추가 검색 결과를 바탕으로 매매 결정 단계로 넘어간다.
